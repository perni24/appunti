# Rust 2018, 2021 e 2024

## Sintesi

Le edition Rust sono modalita di compatibilita del linguaggio. Permettono al linguaggio di evolvere introducendo cambiamenti sintattici o semantici mirati senza rompere automaticamente codice esistente.

Le edition non sono versioni del compilatore. Un progetto dichiara l'edition in `Cargo.toml`, mentre il compilatore Rust continua a essere installato come toolchain.

## Quando usarlo

- Quando crei un nuovo progetto e devi scegliere l'edition.
- Quando mantieni una crate vecchia e valuti una migrazione.
- Quando incontri differenze di comportamento tra crate in edition diverse.
- Quando vuoi capire compatibilita tra dipendenze Rust.

## Come funziona

Nel manifest:

```toml
[package]
edition = "2024"
```

Crate diverse nello stesso dependency graph possono usare edition diverse. Cargo e rustc le compilano insieme.

L'edition influenza il modo in cui il compilatore interpreta il codice di quella crate, ma non crea ecosistemi separati.

## API / Sintassi

Edition in `Cargo.toml`:

```toml
[package]
name = "app"
version = "0.1.0"
edition = "2024"
```

Migrazione assistita:

```bash
cargo fix --edition
```

Controllo:

```bash
cargo check
cargo test
```

## Esempio pratico

Un workspace puo avere una crate legacy in edition 2021 e una nuova crate in edition 2024:

```toml
[package]
edition = "2021"
```

e:

```toml
[package]
edition = "2024"
```

Le due crate possono dipendere l'una dall'altra normalmente se le API sono compatibili.

## Varianti

- Rust 2018: edition che ha consolidato moduli moderni, path e async ecosystem iniziale.
- Rust 2021: miglioramenti incrementali su prelude, closure capture e coerenza del linguaggio.
- Rust 2024: nuova edition con cambiamenti di compatibilita mirati e migrazione guidata.
- Crate mixed-edition: dipendenze con edition diverse nello stesso grafo.
- Migration edition: processo assistito da `cargo fix --edition`.

## Errori comuni

- Pensare che edition e versione del compilatore siano la stessa cosa.
- Credere che tutte le dipendenze debbano usare la stessa edition.
- Migrare edition senza test completi.
- Confondere edition con MSRV.
- Usare la nuova edition come sostituto di una revisione API.

## Checklist

- Il nuovo progetto usa l'edition piu recente supportata dalla tua MSRV?
- La MSRV dichiarata supporta l'edition scelta?
- Hai eseguito `cargo fix --edition` prima della migrazione?
- Hai testato tutte le feature dopo la migrazione?
- Le differenze edition sono documentate se il workspace e misto?
